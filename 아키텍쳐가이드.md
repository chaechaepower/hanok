## 2. CI/CD 파이프라인 (Jenkins)

### 트리거
- **브랜치**: `origin/master` push 시 자동 실행
- **Jenkins 환경변수**:
  - `IMAGE_NAME` = `kimkangyeon/hanok-backend`
  - `COMPOSE_FILE` = `infra/docker-compose.yaml`
  - `ENV_FILE` = `infra/.env.prod`

### 파이프라인 단계

```
[Stage 1] Build (병렬 실행)
     ├── Backend Build
     │    └── ./gradlew bootJar sonar --no-daemon -x test
     │         └── SonarQube 분석: http://j14d105.p.ssafy.io:9000
     └── Frontend Build
          ├── cp .env.fe → .env
          ├── npm install --legacy-peer-deps
          ├── npm run build  (Vite → fe/dist/)
          └── gzip -k dist/landging_model.glb

[Stage 2] Docker Build & Push
     ├── docker build -t kimkangyeon/hanok-backend:prod
     │                -t kimkangyeon/hanok-backend:{BUILD_NUMBER}
     ├── docker push prod 태그
     ├── docker push {BUILD_NUMBER} 태그
     └── 오래된 로컬 이미지 자동 정리 (최신 3개 제외)

[Stage 3] Deploy (Blue/Green 무중단 배포)
     ├── 1. Nginx 미기동 시 자동 기동
     ├── 2. .env.prod, livekit.yaml 파일 생성/갱신
     ├── 3. 인프라 컨테이너 기동 (mysql, redis, redis-exporter, livekit,
     │       prometheus, grafana, loki, promtail, influxdb)
     ├── 4. upstream.conf 분석 → 비활성 타겟 결정
     │        현재 prod 활성 → green 배포
     │        현재 green 활성 → prod 배포
     ├── 5. docker-compose pull + up (비활성 컨테이너)
     ├── 6. 헬스체크 (최대 30회 × 10초 = 5분)
     │        GET http://172.26.0.24:{TARGET_PORT}/actuator/health → 200 대기
     ├── 7. 성공 시:
     │        upstream.conf 갱신 → nginx -s reload → 구버전 stop
     │    실패 시:
     │        새 컨테이너 stop → exit 1 (기존 서버 유지)
     └── 8. FE dist → /var/www/hanok/ 복사
```

### Blue/Green 배포 상태 전환

```
upstream.conf (파일 내용으로 현재 활성 판단)
─────────────────────────────────────────
upstream backend { server backend-prod:8080;  }  ← prod 활성
upstream backend { server backend-green:8080; }  ← green 활성 (현재 상태)
```

| 항목 | backend-prod | backend-green |
|------|-------------|---------------|
| 포트 | :8080 | :8081 |
| 헬스체크 URL | `http://172.26.0.24:8080/actuator/health` | `http://172.26.0.24:8081/actuator/health` |
| DockerHub 이미지 | `kimkangyeon/hanok-backend:prod` | (동일 이미지 재사용) |

## 5. 인증 플로우

### FE Axios 인스턴스 (`fe/src/api/instance/index.ts`)

```
baseURL: VITE_API_BASE_URL
timeout: 20,000ms
withCredentials: true  ← Cookie 자동 전송

Request Interceptor:
  localStorage.accessToken → Authorization: Bearer {token}

Response Interceptor (401 처리):
  1. refreshToken(localStorage) 존재 확인
  2. POST /v1/auth/refresh (withCredentials: true)
      요청: { refreshToken } + Cookie: refreshToken
  3. 응답: 신규 accessToken → localStorage 저장
  4. 원본 요청 Authorization 헤더 교체 후 재시도
  5. Refresh 실패 → localStorage 초기화 + /login 리다이렉트
```

### 엔드포인트: `POST /api/v1/auth/**`

| 메서드 | 경로 | 인증 | 요청 Body | 응답 |
|--------|------|------|-----------|------|
| GET | `/check-email` | ❌ | `?email=` | `void` |
| POST | `/signup` | ❌ | `{email, password, name, phone, birth}` | `{userId, email}` |
| POST | `/identity-verification` | ❌ | `{impUid}` (PortOne 토큰) | `{name, phone, birth, isVerified}` |
| POST | `/login` | ❌ | `{email, password}` | Header: `Authorization: Bearer {accessToken}`, Cookie: `refreshToken` (HttpOnly/Secure/7일), Body: `{accessToken, refreshToken}` |
| POST | `/logout` | ✅ | - | Cookie `refreshToken` 삭제, AccessToken 블랙리스트 |
| POST | `/refresh` | Cookie | Cookie: `refreshToken` | Header: 신규 `Authorization`, Cookie: 신규 `refreshToken` |

## 7. 실시간 통신 레이어 (WebSocket / SSE)

### 7-1. STOMP WebSocket (실시간 경매 · 채팅)

**연결**: `ws(s)://j14d105.p.ssafy.io/ws-connect`  
**프로토콜**: STOMP over WebSocket  
**인증**: STOMP CONNECT 프레임 헤더 `authorization: Bearer {accessToken}`

```
WebSocket Config:
  endpoint:       /ws-connect
  broker:         /broadcast, /private  (SimpleBroker)
  app prefix:     /app
  user prefix:    /user
  msg size limit: 64KB
  buffer limit:   512KB
  send timeout:   20s
  thread pool:    core=200, max=500, queue=1000
```

**클라이언트 → 서버 (SEND)**:
```
SEND
destination: /app/streams/{streamId}
content-type: application/json

{
  "type": "BID | AUCTION_START | CHAT | ...",
  "data": { ... }
}
```

**서버 → 클라이언트 (SUBSCRIBE)**:

| 구독 경로 | 대상 | 이벤트 예시 |
|-----------|------|------------|
| `/broadcast/streams/{streamId}` | 해당 스트림 전체 | BID 결과, AUCTION_START/END, CHAT, VIEWER_COUNT |
| `/private/streams/{streamId}` | 낙찰자 개인 | 낙찰 확정, 에스크로 생성 알림 |

**주요 이벤트 타입 (StreamEventType)**:

| type | 방향 | 주요 data 필드 |
|------|------|---------------|
| `BID` | C→S | `{itemId, bidAmount}` |
| `AUCTION_START` | S→C | `{itemId, startPrice, auctionType, endTime}` |
| `AUCTION_END` | S→C | `{itemId, winnerId, finalPrice}` |
| `CHAT` | 양방향 | `{message, userId, nickname, createdAt}` |
| `VIEWER_COUNT` | S→C | `{count}` |
| `ERROR` | S→C | `{code, message}` |

### 7-2. SSE (서버 → 클라이언트 알림 Push)

**연결**: `GET /api/v1/sse/subscribe`  
**Content-Type**: `text/event-stream`  
**인증**: JWT 필수  
**Nginx**: `proxy_buffering off`, `Connection: ''`, `proxy_read_timeout 3600`

```
클라이언트                    Nginx               Spring Boot BE
    │                          │                        │
    ├─GET /api/v1/sse/subscribe─►                       │
    │                          ├─────proxy──────────────►│
    │                          │                        │ SseEmitter 등록
    │◄─── event: data (Push) ──┤◄────────────────────────┤
    │◄─── event: data (Push) ──┤◄── 알림 발생 시 push ───┤
    │                          │   (팔로우, 낙찰, 공지)   │
```

**SSE 이벤트 타입 (SseEventType)**: 팔로우 알림, 입찰/낙찰 결과, 공지사항, 테스트(TEST)
